## ① 변경 요약

### `luida-core/handoff.rs` (신규)
- `HandoffBundle { schema, origin_machine, exported_at, campaign, quests }` — 운반 단위(JSON). 의존성은 quests 내 **인덱스**로 표현(id 비의존)
- `suspend_campaign(conn, id, machine, force)` — campaign+quests 봉인 + `handoff_state=suspended`/owner. 다른 기기 소유 중이면 거부(force 우회)
- `resume_bundle(conn, bundle, machine)` — 새 campaign으로 import(원자 tx), id 재매핑, in-flight(running/needs_input)→pending 리셋, 의존성 재구성, `handoff_state=resumed`/owner
- `machine_id()` — `LUIDA_MACHINE_ID` > `HOSTNAME` > "local"
- `to_json`/`from_json` (schema 검증)

### `luida-cli`

## ② 설계 의사결정
- **데이터 번들만 운반** (spec §14.7): tavern.db 직접 동기화 금지(WAL 손상). 진행상태를 JSON으로. 미커밋 코드의 git patch transport는 별도 실환경 계층(후속).
- **의존성 = 인덱스**: 기기 간 quest id가 달라지므로 bundle 내 위치로 표현 → import 시 새 id로 재구성.
- **in-flight 리셋**: 재개 기기에선 worktree가 없으므로 running/needs_input quest를 pending으로 되돌려 재디스패치(완료/실패는 보존). branch도 pending이면 초기화.
- **single owner 잠금**: suspended+다른 owner면 재중단 거부(force). resume은 새 owner로 표시.
- **import는 원자 tx**: campaign+quests+deps를 한 트랜잭션으로 → 부분 import 방지.
