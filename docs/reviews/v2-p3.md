## ② 설계 의사결정

- **다중 의존성을 위해 `quest_deps` 조인 테이블 도입**: 0002의 단일 `depends_on_quest_id`로는 다이아몬드(d→b,c) 같은 DAG를 표현 못 해 잘못된 ready 판정이 났다. 조인 테이블이 정답. 단일 컬럼은 "대표 의존"으로 back-compat 유지(기존 테스트/단순 케이스), ready 판정은 **둘 다** 본다.
- **위상순 insert**: 의존성 quest id를 참조하려면 먼저 insert돼야 함 → `validate`가 돌려준 위상정렬 순서대로 insert하고 key→id 매핑 누적.
- **LLM 출력 파싱 견고화**: `extract_json_object`로 첫 `{`~마지막 `}` 추출 → 코드펜스/산문에 감싸여도 파싱. 계획은 `outcome.summary`에서 회수.
- **순차 실행 (동시 한도 미적용)**: 미해결 질문 #3(DAG 동시 실행 한도)은 자원·비용·머지충돌 정책이 필요 → 우선 순차로 안전하게. 병렬·한도는 후속.
- **완료 마감은 report 단계로**: 모든 quest completed여도 `run_campaign`은 status를 `running`으로 둔다. `completed` 전이는 `campaign.report`(Phase D)에서.
