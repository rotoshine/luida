- `AgentInvocation.resume: bool` — `--resume`(맥락 재개) vs `--session-id`(새 세션) 신호
- `EventRepo::latest_payload_for_quest(quest_id, kind)` — 최근 이벤트 payload 조회 (레이어 분리: sidecar가 rusqlite 직접 안 씀)

### `luida-sidecar`
- **dispatch 리팩터**: 실행 공통부 `run_worker` + 판정 `settle_outcome` 추출 → `dispatch_quest`(신규 worktree)·`resume_quest`(기존 worktree + `--resume`) 공유
- **사용자 알림 분리**: dispatcher는 escalation 시 needs_input + 이벤트만 기록(인앱 알림 X). 알림 게이트는 triage/orchestrator (spec §7.4)

  - `notify_user_escalation` — `@user` escalation inmail (dedupe로 멱등)
  - `resume_quest` — needs_input quest를 답변으로 재개
- `MAX_AUTO_RESUME = 2`

## ② 설계 의사결정

- **알림을 dispatcher에서 분리**: spec §7.4 "triage가 사용자 깨울지 결정"을 따라 dispatcher는 신호(needs_input)만, 알림은 triage/orchestrator가 게이트. trivial escalation은 사용자를 안 깨우고 자동 진행.
- **resume = 기존 worktree 재사용**: 신규 worktree를 안 만들고 `quest.worktree_path`에서 `--resume`. 세션 id(`luida-q{id}`)로 직전 맥락 이어받기.
- **자동 재개 횟수 제한(MAX_AUTO_RESUME)**: triage 자동 해소가 무한 escalation 루프에 빠지지 않게 quest당 상한.
- **escalation은 needs_input(좀비 아님)**: headless 프로토콜상 worker가 result 없이 의도적으로 멈춘 신호 → settle_outcome이 needs_input으로(좀비/실패와 구분).
- **알림 멱등(dedupe_key=esc-q{id})**: triage 비통과·cap 도달 등 여러 경로에서 notify 호출돼도 중복 알림 없음.
