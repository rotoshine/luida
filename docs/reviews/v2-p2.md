## ① 변경 요약

신규 crate `luida-sidecar` — 기반 계층(repo·resolver·runtime)을 **실제로 묶어 quest를 굴리는** 계층.

### `worktree.rs`
- `WorktreeProvider` trait + `Worktree { branch, path }`
- `WorktrunkProvider` — `wt switch --create <codename> --base <base> -C <repo> -y` 실행, 경로는 `WORKTRUNK_DIRECTIVE_CD_FILE`로 회수

### `dispatch.rs`
- `dispatch_quest(conn, cfg, quest_id, &dyn WorktreeProvider, runtime_factory)` → `DispatchOutcome`
  1. quest 로드(종료 상태면 거부) → 프로젝트 조회
  2. `resolve("quest.execute", project)` (프로젝트별 override 반영)
  3. worktree 생성 → `set_worktree`/`status=running` + `quest_dispatched` 이벤트
  4. `AgentInvocation`(brief + escalation 규약 주입, cwd=worktree, session_id=`luida-q{id}`) → factory 런타임 실행
  5. 스트림 이벤트 → events 기록(`tool_use`/`escalation`) + progress 갱신(char-safe 절단)
  6. 판정: escalation→`needs_input`(+`@user` escalation inmail), success→`completed`, else→`failed`
- `ESCALATION_PROTOCOL` 상수 (worker brief에 주입하는 `<<LUIDA_ASK>>` 규약)

## ② 설계 의사결정

- **`wt c` 대신 `wt switch --create`**: `wt c`는 `--execute=claude` 포함(interactive REPL 자동 기동). headless는 worker를 Luida가 직접 spawn하므로 worktree 생성만 필요 → `wt switch --create` 사용. 둘 다 worktrunk이므로 **raw git worktree 금지 규약 준수**.
- **runtime factory 주입**: `dispatch_quest`가 `Fn(&ResolvedAgent) -> Result<Box<dyn AgentRuntime>>`를 받음 → 실제 CLI 없이 `ScriptedRuntime`으로 성공/escalation/실패 전 경로 테스트. 실배선은 CLI/server가 `luida_runtimes::runtime_for_kind`로 주입.
- **escalation 우선 판정**: outcome에 escalation이 있으면 success 여부와 무관하게 `needs_input`. (worker가 묻고 멈춘 케이스를 완료로 오인 방지)
- **escalation inmail은 `@user` broadcast + kind=escalation**: dispatch만 broadcast 거부되므로 escalation은 허용. dedupe_key `esc-q{id}`로 같은 quest 중복 알림 차단.
- **progress = 첫 줄 char-safe 절단**: 멀티바이트(CJK) 경계 보존.
