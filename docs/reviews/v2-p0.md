## ① 변경 요약

v2 Rust 워크스페이스 부트스트랩. v1(TS)은 `git tag v1-typescript`로 보존.

```
luida/
├── Cargo.toml                        (workspace: crates/*)
├── crates/
│   ├── luida-core/
│   │   ├── Cargo.toml                (rusqlite bundled, serde, anyhow)
│   │   ├── migrations/0001_init.sql  (projects 테이블)
│   │   └── src/{lib,db,models,repo}.rs
│   ├── luida-tui/                    (ratatui + crossterm — 등록부 골격)
│   │   ├── Cargo.toml
│   │   └── src/lib.rs
│   └── luida-cli/                    (clap — 단일 바이너리 `luida`)
│       ├── Cargo.toml
│       └── src/main.rs
└── (packages/* TS는 그대로 — 레퍼런스)
```

기능:
- `luida db init` — tavern.db 초기화·마이그레이션
- `luida project add <name> --path <p> [--base <b>] [--desc <d>]` — 모험지 등록(upsert)
- `luida project list` / `luida project remove <name>`
- `luida ui` — ratatui 등록부 화면 (모험지 목록 + 빈 상태, j/k 이동, q 종료)

## ② 설계 의사결정

### 마이그레이션: include_str! 임베드 + IMMEDIATE tx
- `MIGRATIONS` 배열에 `include_str!`로 SQL 임베드 → 단일 바이너리에 포함 (TS의 파일 경로 의존 문제 원천 차단)
- 각 마이그레이션은 `unchecked_transaction`으로 atomic 적용, `schema_migrations`로 idempotent

### v2 스키마는 fresh 시작 (v1 호환은 후속)
- v2-P0은 `projects`만. v1의 0001/0002(adventurers/quests/inmail/…)는 재현하지 않음
- "v1 tavern.db 데이터 호환"은 별도 마이그레이션 도구로 후속 (ADR 위험완화 R) — P0는 fresh db 전제
- 후속 Phase에서 quests/campaigns/inmail/events 마이그레이션 추가

### rusqlite bundled
- SQLite를 소스로 번들 → 시스템 sqlite 의존 0, 단일 바이너리 자족
- 첫 빌드 ~14s (이후 증분 빌드 빠름)

### ProjectRepo: lifetime 借用 패턴
- `ProjectRepo<'a> { conn: &'a Connection }` — Connection을 빌려 사용. TS의 Repos facade보다 Rust다운 소유권
- `add`는 `ON CONFLICT DO UPDATE` upsert

### TUI: ratatui 0.29 + crossterm 0.28
- `App` 상태(projects + ListState) + draw/run_loop 분리
- raw mode + alternate screen, 종료 시 복원 보장 (init/restore 함수)
- DQ 풍 색 (gold/dim/green) 인라인 상수

### Major (5건) — **4건 수정 / 1건 설계 deferral**
| # | 이슈 | 처리 |
|---|---|---|
| M1 | `unchecked_transaction` 사용 | **수정** — `migrate(&mut Connection)` + `conn.transaction()` |
| M2 | WAL 다중 writer 직렬화 — busy_timeout만으로 불충분 | **설계 deferral** — P0는 수동 조작이라 무관. v2-standalone.md에 "writer 단일 프로세스 또는 backoff 재시도" 정책 명시 예정(에이전트 오케스트레이션 도입 Phase) |
| M3 | migrate 조회가 `filter_map(.ok())`로 에러 삼킴 → 재적용 위험 | **수정** — `for r in rows { set.insert(r?) }` 전파 |
| M4 | 마이그레이션 부분 적용 가능성 | **확인** — 현재 각 마이그레이션이 (SQL+기록) 단일 tx라 OK. 다건 누적 시 부분적용 인지 — 후속 마이그레이션 추가 시 주의 |

- ✅ m5: 미사용 `thiserror` 제거 + "core는 향후 thiserror로 CoreError enum 타입화" 주석 명시

## ⑤ 다음 Phase 영향 + ADR early-exit 판정

### 🟢 ADR-0001 early-exit 판정: GREEN — Rust 진행
ADR §5 위험완화: "V2-P0를 Rust로 먼저 만들어 개발 속도 실측". 결과:
- **개발 속도**: workspace+core+cli+tui+테스트를 한 번에 작성 → **첫 빌드 13.8s, clippy 0 warning, 7 test pass, CLI 즉시 동작**. borrow checker와 싸운 시간 거의 없음 (스코프가 명확해서)
- **단일 바이너리**: release 2.8M (strip+lto). Bun 런타임 의존 0 → 노트북 2대 hand-off 배포에 이상적
- **결론**: Rust 전환의 개발속도 우려가 이 스코프에선 현실화되지 않음. **Rust로 계속 진행 권장.**
